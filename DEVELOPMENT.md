# mtc — Development Guide

Reference for continuing development on **mtc**, a Tauri v2 + Svelte 5 desktop app
that runs multiple `claude`/`codex` sessions inside WSL on Windows.

> Read this with `README.md` (user-facing) open. This file is the
> engineering/architecture reference.

---

## 1. Tech stack

| Layer | Choice | Why |
| --- | --- | --- |
| Shell | Tauri v2 (Rust) | Native window + small binary + Rust PTY access |
| PTY | `portable-pty` 0.8 | ConPTY on Windows, openpty on Unix — one API |
| Frontend | Svelte 5 (runes) + TypeScript + Vite | Light, reactive, fast build |
| Terminal | `@xterm/xterm` 5 + `@xterm/addon-fit` | Full VT rendering in the webview |
| Persistence | JSON file in app-config dir (atomic write) | No DB needed for MVP |

Target runtime: **Windows + WSL**. Developed/verified on **Linux/WSL** (see §7).

## 3. Data flow

### Spawning a terminal
1. User clicks a profile chip in `CenterPanel` → `addPane(profile, 'v')`.
2. A `TerminalPane` mounts: creates xterm + FitAddon, `fit()`, then
   `api.createSession(profile, cols, rows, {onData, onExit})`.
3. `api.createSession` makes a `Channel<PtyMessage>` and `invoke('create_session', …)`.
4. Rust `create_session` → `PtyManager::create`:
   - `build_invocation(profile, cfg!(windows))` → `(program, args, cwd)`.
   - `portable-pty` opens a PTY, spawns the command, drops the slave.
   - A reader thread pumps PTY bytes → base64 → `channel.send(Data{…})`; on EOF it
     reaps the child and sends `Exit{code}`.
5. Frontend channel `onmessage`: `data` → `term.write(bytes)`, `exit` → mark exited.
6. `term.onData` → `api.writeSession`; `ResizeObserver` → `api.resizeSession`.

### Profile → process (the core platform split) — `profile.rs::build_invocation`
- **Windows:** `wsl.exe [-d <distro>] [--cd <cwd>] -- bash -lic "<command>; exec bash -l"`
  (no `--` tail when command empty → plain interactive login shell).
- **Unix (dev/CI):** `bash -lic "<command>; exec bash -l"` (or `bash -l`), `cwd` honored.
  This is what lets the whole PTY path be exercised on Linux without Windows.

### Persistence
- `state.svelte.ts` holds `AppData` in a `$state`. Every mutation calls a **250ms
  debounced** `saveAppData($state.snapshot(data))` → `save_app_data` writes
  `app-data.json` atomically (temp + rename) in the OS app-config dir.
- On boot, `app.init()` calls `load_app_data`; if it throws (no Tauri backend, i.e.
  plain `vite` in a browser) it sets `standalone = true` and uses defaults.

## 4. Gotchas & learnings (don't re-discover these)

- **`portable-pty` `Child` must be reaped.** `close()` calls `kill()` **then `wait()`**
  to avoid Unix zombies. The natural-exit path reaps in the reader thread. Keep both.
- **Mutex poisoning.** `pty.rs` uses a `lock()` helper that recovers from a poisoned
  mutex (`unwrap_or_else(|e| e.into_inner())`) so one bad session can't disable all
  terminals app-wide. Don't revert to `.lock().unwrap()`.
- **`bus.send` is last-writer-wins.** The active pane registers its sender in
  `bus.send`; `TerminalPane.onDestroy` clears it if it still owns it. If you add
  multiple senders or richer routing, prefer keying by `focusedId` in `CenterPanel`.
- **Maximize must NOT unmount panes.** `CenterPanel` derives `order` from the tile
  `tree` only and hides non-maximized panes with `display:none`. If you switch to
  rendering `effectiveTiles` directly, hidden panes get removed from the `{#each}`,
  their `TerminalPane` unmounts, and **the backend session is killed**. Don't.
- **Tauri IPC arg casing.** Rust snake_case params map to JS camelCase automatically
  (`on_event` ↔ `onEvent`). `Profile` uses `#[serde(rename_all = "camelCase")]` to
  match the TS type, so its fields must stay in sync on both sides.
- **`tauri = { features = [] }` does NOT disable defaults.** Defaults are additive
  unless you set `default-features = false`. Leave it.
- **Svelte 5 reactive state over IPC.** Pass `$state.snapshot(...)` (a plain deep
  clone) to `invoke`, never the live proxy.
- **Custom commands need no capability entry**; only plugin commands do. `dialog:default`
  is in `capabilities/default.json` for the folder picker.
- **Icons are required to build.** Regenerate with `node scripts/gen-icon.mjs &&
  npm run tauri icon app-icon.png`. The generated `src-tauri/icons/` is committed.

## 5. How to extend

### Add a Tauri command
1. Write the function in the relevant `src-tauri/src/*.rs` module.
2. Add a thin `#[tauri::command]` wrapper in `main.rs` and register it in
   `tauri::generate_handler![…]`.
3. Add a typed wrapper in `src/lib/api.ts`.
4. If it touches a plugin/core capability, grant it in `capabilities/default.json`.

### Add persisted state
1. Extend `AppData` in `types.ts` and `normalizeAppData` in `defaults.ts`.
2. Add actions on `AppState` in `state.svelte.ts` (call `scheduleSave()`).
   The Rust side is schema-agnostic (`serde_json::Value`), so no backend change needed.

### Ideas / next steps (not yet built)
- Persist & restore the session layout across restarts.
- Per-profile env vars and a distro picker populated from `wsl -l -q`.
- "Scan skills via WSL" (`wsl.exe -d <distro> -- bash -lc …`) instead of UNC paths.
- Scrollback search, copy/paste affordances, font-size controls in xterm.
- Tab the bottom of a pane to show exit status / restart button.

## 6. Verification (run before committing)

```bash
# frontend
npm run test        # vitest — tiling (8 tests)
npm run check       # svelte-check — 0 errors expected
npm run build       # type-check + vite build

# backend (in src-tauri/)
cargo test                       # profile invocation, skill scan, storage (9 tests)
cargo run --example pty_smoke    # confirms portable-pty streams + resizes

# full app (Linux headless sanity — boots to event loop, no panic)
xvfb-run -a ./src-tauri/target/debug/mtc   # Ctrl-C to stop
```

On Windows: `npm run tauri dev` (the only way to exercise WebView2 + real `wsl.exe`).

## 7. Verification limits (important)

Development happens in a **Linux/WSL** environment. Verified there: frontend
(type-check, build, vitest), Rust (`cargo test`, `cargo build`), `portable-pty`
streaming against `bash`, and a headless `xvfb` boot (no panic, valid Tauri config).

**Not verifiable on Linux** — must be checked on a Windows host with `npm run tauri dev`:
- WebView2 window rendering.
- `wsl.exe` spawning and the WSL `--cd` / `-d <distro>` arguments.
- The Windows folder picker (dialog plugin) and `\\wsl.localhost\…` skill roots.

When you change `pty.rs` spawn logic or `profile.rs` Windows branch, re-test on Windows.
