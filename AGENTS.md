# mtc — Multi-Terminal Claude Code

## Purpose
A Tauri v2 desktop app that orchestrates multiple WSL terminal sessions running
`claude`, `codex`, or plain shells, tiled side-by-side in one window. The Svelte 5
frontend renders the UI and xterm.js terminals; the Rust backend spawns PTYs (via
`portable-pty`), streams their output to the frontend over Tauri channels, scans the
filesystem for Claude skills, and persists app data. On Windows it enters WSL via
`wsl.exe`; on Unix it runs `bash` directly so the same plumbing can be developed and
tested without a Windows host.

## For AI Agents

### Working In This Directory
- This is a **hybrid Tauri app**: frontend (`src/`, TypeScript/Svelte) and backend
  (`src-tauri/`, Rust) are separate build pipelines that meet at the Tauri IPC
  boundary. A change to a command's signature must be applied on **both** sides:
  the Rust `#[tauri::command]` in `src-tauri/src/main.rs` and the `invoke(...)`
  wrapper in `src/lib/api.ts`. Serde uses `camelCase` rename, so Rust `snake_case`
  fields map to frontend camelCase (e.g. `keep_open` ↔ `keepOpen`).
- The app must run in two modes: **inside Tauri** (real WSL terminals) and
  **plain browser** (`vite` only — no backend). The frontend degrades gracefully
  to a "standalone" mode; preserve that fallback when touching state/API code.

### Testing Requirements
- Frontend: `npm test` (vitest) — currently covers `src/lib/tiling.ts`.
- Backend: `cargo test` inside `src-tauri/` — covers `profile.rs` and `storage.rs`.
- PTY smoke check: `cargo run --example pty_smoke` inside `src-tauri/`.
- Type check before build: `npm run check` (svelte-check).

### Common Patterns
- Svelte 5 **runes** (`$state`, `$derived`, `$effect`, `$props`) throughout — not
  the legacy store API. Reactive singletons live in `.svelte.ts` files.
- IPC bytes are base64-encoded over channels and decoded in `src/lib/api.ts`.

## Dependencies

### External

- **portable-pty 0.8** (Rust) — cross-platform PTY spawning.
- **vitest** — frontend unit tests.
